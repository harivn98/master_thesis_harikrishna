1. PROJECT OVERVIEW
================================================================================

This master thesis project implements an end-to-end pipeline for generating and
editing lane markings in autonomous vehicle camera footage using state-of-the-art
video inpainting models, and evaluating the impact of edited lane markings on
autonomous driving trajectory prediction using Vision-Language-Action (VLA) models.

The system combines SAM2 (Segment Anything Model 2) for road segmentation,
VideoPainter (SIGGRAPH 2025) for video inpainting, and Alpamayo-R1-10B for
VLA-based trajectory prediction — all orchestrated through a unified three-stage
master pipeline with extensive custom modifications for autonomous driving
applications.

PRIMARY OBJECTIVES:
- Generate synthetic lane marking datasets from Physical AI autonomous vehicle data
- Enable precise control over lane marking attributes (count, color, pattern)
- Train specialized FluxFill LoRA models for lane marking generation
- Evaluate inpainting quality using VLM-based assessment and numerical metrics
- Assess impact of lane marking modifications on autonomous driving trajectory
  prediction using Alpamayo-R1-10B VLA model
- Provide end-to-end pipeline orchestration (SAM2 → VideoPainter → Alpamayo)

KEY TECHNOLOGIES:
- SAM2.1 (Hiera-Large) for video segmentation
- VideoPainter with CogVideoX-5B backbone
- FluxFill image inpainting model
- Qwen2.5-VL Vision Language Model (7B/72B variants)
- Alpamayo-R1-10B Vision-Language-Action model for trajectory prediction
- Google Cloud Platform (GCS) with HLX workflow orchestration
- Master pipeline orchestrator for three-stage sequential execution

MASTER PIPELINE ORCHESTRATION (workflow_master.py)
──────────────────────────────────────────────────
All three inference stages are chained in a single HLX workflow with
a shared run_id for end-to-end traceability:

  SAM2 ──(run_id)──▶ VideoPainter ──(gcs_path)──▶ Alpamayo

  =========  ====================================================
  Stage      GCS output
  =========  ====================================================
  SAM2       gs://…/outputs/sam2/<run_id>/
  SAM2 (VP)  gs://…/outputs/preprocessed_data_vp/<run_id>/
  VP         gs://…/outputs/vp/<run_id>/
  Alpamayo   gs://…/outputs/alpamayo/<run_id>/
  =========  ====================================================

Each stage runs in its own container image (heavy ML deps are isolated)
while the workflow graph is serialised from a lightweight orchestrator
container. Data-dependency edges enforce strict sequential execution.

MULTI-CAMERA INPUT & BLACK-OUT STRATEGY:
────────────────────────────────────────
The Alpamayo-R1-10B VLA model natively operates on a 7-camera multi-view
input, each providing 4 consecutive frames.  The cameras, their indices,
and approximate horizontal FOVs are:

  Index  Camera Name                    FOV
  ─────  ─────────────────────────────  ────
    0    camera_cross_left_120fov       120°
    1    camera_front_wide_120fov       120°
    2    camera_cross_right_120fov      120°
    3    camera_rear_left_70fov          70°
    4    camera_rear_tele_30fov          30°
    5    camera_rear_right_70fov         70°
    6    camera_front_tele_30fov         30°  ← target camera

In this thesis, only camera_front_tele_30fov (index 6) is edited by
VideoPainter.  To isolate the effect of the edited front camera on the
model's trajectory prediction, the "black_non_target_cameras" flag is
enabled by default (ALPAMAYO_BLACK_NON_TARGET_CAMERAS=true).

When enabled, the inference pipeline:
  1. Loads ALL 7 camera frames from the PhysicalAI-AV dataset
  2. Replaces camera_front_tele_30fov frames with VP-edited frames
  3. Sets ALL OTHER 6 camera tensors to zero (black frames)
  4. Feeds the resulting 7-camera tensor (28 frames total) to the model

This means the model still receives the expected (N_cameras, 4, 3, H, W)
input tensor, but only the target camera contains meaningful visual
information.  The remaining 6 cameras are entirely black.

Rationale: Without blacking out the other cameras, the model could rely
on the original (unedited) front-wide or cross-camera views to predict
the trajectory, potentially masking the effect of the VP-edited lane
markings on the front tele camera.  By zeroing the non-target cameras,
the model must base its visual understanding solely on the generated
front camera video.

Note: The ego history (ego_history_xyz, ego_history_rot) is always
sourced from the real driving data and is identical for both the
"original" and "generated" inference passes.  This means ego motion
acts as a confounding variable — the model can partially infer the
trajectory from deceleration/acceleration patterns in the ego history,
independent of the visual input.

MODEL INPUTS:
  The model receives three inputs for each inference pass:
    1. tokenized_data  — VLM-processed camera images
       Shape: 7 cameras × 4 frames = 28 images (each 1080×1920)
    2. ego_history_xyz — past 16 ego positions in ego frame (x=fwd, y=left, z=up)
       Shape: (16, 3)
    3. ego_history_rot — past 16 ego rotation matrices
       Shape: (16, 3, 3)

  The model outputs:
    - pred_xyz:  (S, 64, 3) — S sampled trajectory predictions, 64 future waypoints
    - pred_rot:  (S, 64, 3, 3) — corresponding rotation matrices
    - extra:     dict containing chain-of-thought reasoning traces

DUAL-PASS INFERENCE:
  For each video, two inference passes are performed:
    Pass 1 (ORIGINAL):  Uses unmodified real camera frames from PhysicalAI-AV
    Pass 2 (GENERATED): Uses VP-edited frames on the target camera + blacked-out others
  Both passes use the same ego history, enabling direct comparison of how
  the visual edits affect trajectory prediction (minADE metric).
  Configuration: black_non_target_cameras=true (non-target cameras are
  zeroed out to isolate the effect of the edited front camera).

TRAJECTORY PROJECTION ONTO CAMERA IMAGE:
  The 3D trajectories (in ego frame) are projected onto the 2D camera
  image using a pinhole camera model:

  1. Ego-to-camera coordinate transform:
       x_cam = −y_ego     (ego left → camera right)
       y_cam = −z_ego     (ego up → camera down)
       z_cam =  x_ego     (ego forward → camera depth)
     For non-front cameras, an additional yaw rotation is applied first
     based on the camera's heading angle.

  2. Ground-plane projection:
     To avoid vertical discrepancy between GT (which has real road-slope z)
     and predictions (which output z=constant from the unicycle model),
     all trajectories are projected onto a fixed ground plane at
     camera_height = 1.5 m.  This ensures both GT and predictions appear
     on the road surface in the image.

  3. Perspective projection:
       f_x = (W / 2) / tan(FOV / 2)
       u = f_x × x_cam / z_cam + W/2
       v = f_x × y_cam / z_cam + H/2
     Points behind the camera (z_cam ≤ 0.5 m) are filtered out.

  4. Progressive reveal:
     Trajectories are drawn progressively — each video frame maps to a
     linearly interpolated waypoint index:
       t_reveal = (frame_idx / (num_frames − 1)) × (64 − 1)
     Only waypoints 0..t_reveal are drawn on that frame, creating an
     animation of the trajectory being "drawn" forward over time.

OVERLAY VIDEO (_overlay.mp4):
  The individual overlay video shows only the VP-edited video with:
    • Red line:   Ground truth trajectory
    • Cyan line (BGR: 255,255,0): Predicted trajectory (sample #1)
    • Additional samples cycle through green, orange, yellow, magenta
    • Gray line:  Ego history (past path)
    • BEV inset:  250×250 px in bottom-left corner
    • Legend box: Semi-transparent in top-right corner
    • HUD text:   minADE, clip_id, camera info from JSON

Caption Generation Details:

    "You are a driving scene labeling assistant. You only describe lane
     line markings on the road surface. First, check whether the segmented
     region shows a CLEAR road surface with lane line markings clearly

    The Qwen 2.5-VL-7B model may misclassify lane attributes — e.g.,
    labelling a faded or partially occluded marking as "single solid

    visual-text associations. A manual audit was not conducted.

Pipeline Stages:

STAGE 1: Video-to-Dataset Bridge
  1. Parse clip_id and camera_name from VP output filename
     Pattern: <clip_id>.<camera_name>_vp_edit_sample0.mp4
  2. Decode last 4 consecutive frames from VP video using PyAV
  3. Detect VP video fps and compute matching time_step

6. Alpamayo VLA Inference Workflow (NEW)
   File: vla/alpamayo/workflow_alpamayo.py
   Entry: alpamayo_vla_inference_wf
   
   @task Configuration:
     compute: DedicatedNode(A100_80GB_1GPU)
     container_image: [ALPAMAYO_CONTAINER]
     mounts:
       - FuseBucket (Alpamayo checkpoints)
       - FuseBucket (VP output video data)
     environment:
       PYTORCH_CUDA_ALLOC_CONF: "expandable_segments:True"
       HF_HOME: "/root/.cache/huggingface"
       HF_TOKEN: (passed from environment)
   
   Parameters:
     - video_data_gcs_path: str (GCS path to VP edited videos)
     - output_run_id: str
     - model_id: str (default: mounted checkpoint or HF repo)
     - num_traj_samples: int = 1
     - video_name: str = "auto" (filter by stem, or all)
   
   Process:
     1. Symlink FuseBucket checkpoint → /workspace/alpamayo/checkpoints
     2. Discover video files via FuseBucket mount (exclude _generated.mp4)
     3. Load Alpamayo model ONCE (bfloat16, flash-attn)
     4. For each video:
        a. Parse clip_id + camera_name from filename
        b. Decode VP frames (last 4 at native fps)
        c. Load PhysicalAI-AV dataset with matched time_step
        d. Replace camera frames with VP output
        e. Run VLA inference → trajectories + reasoning
        f. Compute minADE vs ground truth
        g. Save JSON + NPZ + overlay video
     5. Write aggregate report
     6. Upload all outputs to GCS via gcsfs

BUILD SCRIPTS

Master Pipeline (scripts/build_and_run.sh):
  1. Builds 4 Docker images: SAM2, VP, Alpamayo, Master
  2. Tags all with shared MASTER_RUN_ID
  3. Pushes to Google Artifact Registry
  4. Exports container image env vars
  5. Submits master_pipeline_wf via hlx wf run
  
  Features:
    - SKIP_BUILD=1 to reuse images
    - Auto-generates MASTER_RUN_ID = RUN_ID + timestamp
    - Prints comprehensive configuration summary
    - SAM2_EXPECTED_VIDEOS calculation for display

Alpamayo Standalone (vla/alpamayo/scripts/build_and_run.sh):
  1. Auto-discovers VP output folder matching RUN_ID
     (gcloud storage ls + grep for matching prefix)
  2. Builds and pushes Alpamayo image with RUN_TAG in name
  3. Requires HF_TOKEN for PhysicalAI-AV streaming
  4. Submits alpamayo_vla_inference_wf

Example (Master):
  #!/bin/bash

    ...

EVALUATION METHODOLOGY
──────────────────────
1. Automated VLM Evaluation (Qwen2.5-VL)
   - Semantic correctness (lane attributes match instruction)
   - Visual quality (texture, alignment, lighting)
   - Context preservation (background unchanged)
   - Structured JSON output (verdict + notes)

2. Manual Visual Inspection
   - Sample 100 generated videos
   - Rate on 1-5 scale for:
     * Semantic accuracy
     * Visual realism
     * Temporal consistency
     * Artifact presence

3. Quantitative Metrics (when applicable)
   - First-frame pass rate (VLM evaluation)
   - Average refinement iterations
   - Processing time per video

4. Automated Numerical Video Quality Metrics (NEW)
   Computed per-video via _evaluate_video() in workflow_vp.py using
   MetricsCalculator from evaluate/metrics.py. Two passes per video:
   a) Full-frame metrics — compare entire original vs. generated frame
   b) Masked-region metrics — compare only within inpainted mask area

   Metrics (both full-frame and masked-region):
   - PSNR (Peak Signal-to-Noise Ratio): 10·log10(1/MSE) dB, higher=better
   - SSIM (Structural Similarity Index): luminance+contrast+structure [0,1]
   - LPIPS (Learned Perceptual Image Patch Similarity): deep perceptual
     distance, lower=better
   - MSE (Mean Squared Error): (1/N)Σ(x_i − y_i)², lower=better
   - MAE (Mean Absolute Error): (1/N)Σ|x_i − y_i|, lower=better
   - Temporal Consistency: frame-to-frame coherence, higher=better
   - CLIP Score: text-video alignment (only when caption provided)

   Quality Assessment (from full-frame metrics):
   - PSNR > 40 dB → +1 point
   - SSIM > 0.95  → +1 point
   - LPIPS < 0.1  → +1 point
   - Score ≥ 2: EXCELLENT | Score = 1: GOOD | Score = 0: MODERATE

   Output: eval_{video_id}.txt saved alongside generated video in GCS
   Contains: video metadata, resolution, FPS, mask coverage %,
   full-frame metrics block, masked-region metrics block, quality label.

5. VLA Trajectory Impact Assessment (NEW — Alpamayo)
   - minADE (minimum Average Displacement Error): distance between
     predicted and ground-truth ego-vehicle trajectories (metres)
     Formula: min_s (1/T) Σ_t ||pred_xy^(s)_t − gt_xy_t||_2
   - Computed per-video and aggregated across the run
     (average, best, worst minADE in report)
   - Ground truth from PhysicalAI-AV HuggingFace dataset
   - Per-video compute metrics: inference time, GPU memory, RAM usage
   - Measures whether edited lane markings shift VLA driving behaviour
   - Trajectory overlay visualization with BEV inset for qualitative eval

     p3 (single yellow solid) achieves the highest improvement rate (52%),
     while p5 (dashed white) is worst (40%). Lane marking complexity
     directly impacts VLA trajectory prediction quality.

     ~46% of videos see trajectory improvement, ~54% degrade. The net
     effect is a small average degradation of +0.16 m from the 3.78 m
     baseline. This motivates the LoRA fine-tuning approach (see above)
     to improve inpainting quality and expected trajectory impact.

  3. CLIP SCORE CORRELATES WITH PROMPT COMPLEXITY
     Simpler prompts (p1 single white = 23.2) get higher CLIP alignment;
     complex ones (p4 double yellow = 19.2, p5 dashed = 22.7) score lower.

  4. TEMPORAL CONSISTENCY IS UNIFORMLY EXCELLENT
     All experiments score >0.985, confirming VideoPainter produces
     temporally smooth outputs across all lane-marking styles.

  5. vps90 OFFERS NO ADVANTAGE OVER 100frames
     Slightly worse across all metrics — use the full 100 frames.

  6. CAPTION REFINEMENT IS THE BOTTLENECK FOR p5 (DASHED LINES)
     50% failure rate suggests the VLM-based caption evaluator struggles
     with dashed/intermittent line descriptions, contributing to 60%
     longer generation time for p5.

  7. HEAVY TAIL IN ADE
     The worst minADE is ~32 m across all experiments, indicating a small
     subset of videos where the VLA trajectory deviates massively. These
     outliers dominate the mean; the median ADE (2.6–3.4 m) is a more
     representative metric.

  8. WHEN EDITING HELPS, IT HELPS SUBSTANTIALLY
     The ~46% of improved videos see an average improvement of −0.9 m
     (compared to +1.1 m degradation). Individual videos see up to −8.8 m
     improvement, suggesting lane quality has a strong per-scene effect.

── LORA KEY FINDINGS ──

  1. LORA DOES NOT IMPROVE INPAINTING QUALITY
     SSIM drops from 0.510 to 0.487 and LPIPS increases from 0.458 to
     0.487. PSNR is marginally higher (+0.13 dB). LoRA fine-tuning on
     domain-specific lane marking data did not improve pixel-level or
     perceptual quality metrics over the base FluxFill model.

  2. LORA INCREASES VLA TRAJECTORY DEGRADATION BY 3.8×
     Mean ADE delta rises from +0.16 m (base) to +0.61 m (LoRA) —
     nearly four times worse. LoRA-adapted inpainting introduces more
     downstream trajectory degradation, contrary to the hypothesis
     that domain-specific fine-tuning would improve functional impact.

  2b. ROOT CAUSE: LIKELY FALSE POSITIVES IN LORA TRAINING DATA
     The LoRA training datasets were auto-generated using Qwen 2.5-VL
     captions + semantic attribute filtering. If the Qwen model produced
     inaccurate captions (mislabelling lane types, missing attributes),
     the resulting datasets would contain false positives — images whose
     captions do not match their actual content. Training on mislabelled
     data teaches incorrect text-image associations, directly explaining
     why LoRA degrades rather than improves quality. A manual audit of
     training data quality was not conducted; this is recommended before
     any future LoRA training attempt.

  3. DEGRADATION IS OUTLIER-DRIVEN
     Median ADE delta across LoRA prompts is near zero (+0.04 to
     +0.08 m), while the mean is +0.61 m. A small number of clips
     (approximately 6 of 100 with >5 m degradation) dominate the mean.
     For most videos, LoRA editing has minimal trajectory impact.

  4. CLIP SCORE HIERARCHY IS PRESERVED
     LoRA maintains the same prompt ordering: p1 (22.6) > p5 (22.0) >
     p2 (21.5) > p3 (21.1) > p4 (19.0). This matches the base results
     and confirms that prompt complexity, not model adaptation, drives
     text-video alignment differences.

  5. CAPTION REFINEMENT STILL FAILS FOR DASHED LINES
     p5 pass rate with LoRA is 6/20 (30%) compared to 49/100 (49%)
     without LoRA — worse, not better. LoRA fine-tuning did not resolve
     the dashed-pattern generation failure identified in the base
     experiments. This rules out the hypothesis that the base model's
     inability to generate dashed lines was the primary cause of VLM
     rejection.

  6. 20-VIDEO SAMPLE LIMITS STATISTICAL POWER
     With only 20 videos per prompt (vs. 100 in the base run), per-prompt
     statistics are more sensitive to individual outliers. The LoRA
     results should be interpreted with caution; a larger-scale LoRA
     inference run (100 videos) would provide more robust comparisons.

  7. ALL PROMPTS PRODUCE UNIFORM VP METRICS
     PSNR (11.13 dB), SSIM (0.487) and temporal consistency (0.985)
     are virtually identical across all 5 LoRA prompts — the same
     pattern observed in the base experiments, confirming that pixel-
     level metrics are insensitive to lane content regardless of model.


PROMPT ADHERENCE EVALUATION: HOW WELL DO GENERATED VIDEOS MATCH THE PROMPT?
────────────────────────────────────────────────────────────────────────────
Prompt adherence measures the degree to which the generated video content
matches the text prompt used to instruct the inpainting model. Two
complementary metrics are used:

  Metric 1: CLIP Score (continuous, 0–100 scale)
    Computed by encoding the text prompt and generated video frames through
    CLIP's joint embedding space, measuring cosine similarity. Higher values
    indicate stronger text-video alignment. Captures global semantic
    correspondence (colour, pattern, visual content).

  Metric 2: VLM Caption Refinement Pass Rate (binary, 0–100%)
    The Qwen 2.5-VL evaluator inspects the generated first frame against the
    editing instruction, producing PASS/FAIL based on five dimensions:
    lane count (single/double), colour (white/yellow), pattern (solid/dashed),
    perspective alignment, and context preservation. Aggregated as pass %.

Per-Prompt Prompt Adherence Comparison:

  Prompt   CLIP (Base)  CLIP (LoRA)  Δ      Pass% (Base)  Pass% (LoRA)  Δ
  ------   -----------  -----------  -----  ------------  ------------  -----
  p1         25.38        22.60      −2.78      99%          100%        +1%
  p2         22.29        21.48      −0.81     100%           95%        −5%
  p3         21.61        21.11      −0.50      97%           85%       −12%
  p4         19.41        19.02      −0.39      95%           90%        −5%
  p5         19.93        22.03      +2.10      49%           30%       −19%
  ------   -----------  -----------  -----  ------------  ------------  -----
  Mean       21.72        21.25      −0.47      88%           80%        −8%

Key Findings:
  - LoRA reduces CLIP Score for 4/5 prompts; mean drops 0.47 points
  - LoRA reduces VLM pass rate from 88% to 80% aggregate
  - p5 (dashed) is worst affected: pass rate drops 49% → 30%
  - p5 paradoxically has a HIGHER CLIP Score with LoRA (+2.10) despite
    LOWER pass rate — CLIP's global embedding assigns high similarity
    to partial dashed patterns that fail VLM semantic inspection
  - Both measures confirm: base FluxFill has better prompt adherence
    than LoRA-adapted FluxFill with auto-generated training data

Implication for Downstream Evaluation:
  Prompt adherence directly affects the validity of Alpamayo trajectory
  evaluation. If the generated lane markings deviate from the prompt
  (wrong colour, wrong pattern, wrong count), the measured trajectory
  change reflects VLA response to UNINTENDED stimuli rather than a
  controlled test of the specified lane type. The 8% aggregate drop in
  VLM pass rate with LoRA means approximately 8 additional videos per
  100 contain lane markings that do not match the prompt specification —
  introducing confounds into the trajectory impact analysis.


END-TO-END LORA PIPELINE: DATA GENERATION → TRAINING → INFERENCE
─────────────────────────────────────────────────────────────────
This section documents the complete LoRA pipeline from automated training
data generation through to inference integration in edit_bench.py. The
pipeline comprises five stages, each implemented as a standalone script
connected through GCS and CSV file interfaces.

STAGE 1: AUTOMATED TRAINING DATA GENERATION
  File: segmentation/sam2/data_generation.py (1,432 lines)
  Entry: fluxfill_data_generation_wf (HLX workflow)

  The pipeline generates (image, mask, caption) triplets from raw driving
  videos using a multi-GPU worker architecture:

  1. Video Discovery
     GCS prefix traversal lists MP4 files in the chunk structure
     (chunk_0000 through chunk_0199). Parameters control chunk range,
     files per chunk, and early-stop optimisation.

  2. Frame Extraction
     Six frames per video at fixed positions (frame numbers 1, 100, 200,
     300, 400, 500) using FFmpeg. Widely spaced intervals ensure road
     geometry and lighting diversity.

  3. SAM 2 Mask Generation
     The image predictor mode (single-frame, not VOS) generates a binary
     road mask for each frame using the bottom-half point grid. This
     differs from the VOS mode used for video segmentation because
     temporal consistency is not needed for independent training images.

  4. Qwen 2.5-VL-7B Caption Generation
     Each frame is sent to Qwen2.5-VL-7B-Instruct with a structured
     two-part prompt:

     System Prompt (data_generation.py, line 1062):
       "You are a driving scene labeling assistant. You only describe
        lane line markings on the road surface. First, check whether the
        segmented region shows a CLEAR road surface with lane line
        markings clearly visible. If the road is not clear or the lane
        line markings are not clearly visible, output 'unknown' for ALL
        fields. Return ONLY valid JSON with keys: lane_marking_count,
        lane_marking_pattern, lane_marking_color. Do NOT wrap the JSON
        in Markdown, code fences, or backticks. Use this fixed
        vocabulary: lane_marking_count={single|double|unknown},
        lane_marking_pattern={solid|dashed|mixed|unknown},
        lane_marking_color={white|yellow|mixed|unknown}."

     User Prompt:
       "Look only at the segmented road region. Only answer if it is a
        clear road with lane line markings visible. Classify lane marking
        count (single vs double line), pattern (solid vs dashed), and
        color (white vs yellow)."

     The VLM output is parsed by _lane_prompt_from_qwen_output():
       a. JSON parsing to extract lane_marking_count/pattern/color
       b. Fallback: regex extraction if JSON fails
       c. Final fallback: "no clear road with lane line markings"
       d. Normalise to: "road with {count} {color} {pattern} lane markings"

  5. Dataset Assembly
     CSV with columns: image, mask, prompt, prompt_2
     All paths relative to dataset root. Images and masks stored in
     separate subdirectories. Complete dataset uploaded to GCS.

  Multi-GPU Architecture:
     - multiprocessing.get_context("spawn") for CUDA safety
     - One subprocess per GPU, each with own SAM 2 + Qwen instances
     - Parent→worker task queue, worker→parent result queue
     - Per-device model singletons avoid redundant loading

  Scale: Chunks 0–99 (half the dataset), yielding 13,372+ training samples.

STAGE 2: SEMANTIC ATTRIBUTE FILTERING
  File: segmentation/sam2/filter_fluxfill_dataset.py (751 lines)
  Entry: filter_fluxfill_dataset_wf (HLX workflow)

  Filters the raw dataset by specific lane marking attributes using regex:
    "road with (?P<count>...) (?P<color>...) (?P<pattern>...) lane markings"

  Configuration:
    - count: single | double | unknown | any
    - color: white | yellow | mixed | unknown | any
    - pattern: solid | dashed | mixed | unknown | any
    - CONFIG_REQUIRE_CLEAR_ROAD=1: rejects "unknown" in any attribute

  Five filtered datasets created (one per lane marking type):
    single_white_solid:      1,522 images

    single_white_dashed:       373 images
    ─────────────────────────────────
    Total:                  13,372 images

STAGE 3: LORA TRAINING
  File: generation/VideoPainter/train/train_fluxfill_inpaint_lora.py (606 lines)
  Entry: fluxfill_training_wf (HLX workflow via training_workflow.py)

  Five independent LoRA adapters trained in parallel on A100 80 GB GPUs.

  Architecture:
    Base Model:   FluxFillPipeline (FLUX.1 Fill, black-forest-labs)
    Frozen:       VAE, text_encoder, text_encoder_2, base transformer
    Trainable:    LoRA adapter layers only

    LoRA Config (from peft.LoraConfig):
      r = 16                         # rank
      lora_alpha = 16                # scaling factor
      init_lora_weights = True
      target_modules = ["to_q", "to_k", "to_v", "to_out.0"]
      dropout = 0.0

  Training Loop (flow matching objective):
    For each batch:
      1. Load image, mask, prompt from CSV (InpaintCsvDataset)
      2. Preprocess: image_processor → [B,C,H,W], mask → binary
      3. Masked image: image * (1.0 - mask)
      4. VAE encode: x0 = (encode(image).sample - shift) * scale
      5. Pack latents: _pack_latents(x0, B, C, H_lat, W_lat)
      6. Prepare mask latents: concat(masked_image_latents, packed_mask)
      7. Encode prompts: (prompt_embeds, pooled_embeds, text_ids)
      8. Sample random timestep t ∈ [0, T]
      9. Add noise: xt = scale_noise(x0, t, ε)
     10. Target: target = ε − x0
     11. Forward: pred = transformer(concat(xt, mask_latents), t, ...)
     12. Loss: MSE(pred, target)  [flow matching loss]
     13. Backward with gradient accumulation
     14. AdamW step + cosine LR scheduler step
     15. Checkpoint at interval

  Hyperparameters (all 5 jobs identical):
    Optimiser:              AdamW
    Learning rate:          1e-5
    LR schedule:            Cosine annealing
    Warmup steps:           50
    Epochs:                 5
    Batch size:             1 (per GPU)
    Gradient accumulation:  4 (effective batch = 4)
    Mixed precision:        bfloat16
    Checkpoint interval:    100 steps
    Seed:                   42

  Output:
    - LoRA weights via FluxFillPipeline.save_lora_weights()
    - training_summary.txt (timing, loss, hyperparams, system info)
    - loss_history.json (per-step + per-epoch loss)

  Training Results (Run 20260222_201003):

    ---------------------   ------   ------  ---------   ----------
    single_yellow_solid       372      465   59.1 min      0.1652
    single_white_dashed       373      470   59.0 min      0.1455

LIMITATIONS & KNOWN CHALLENGES
──────────────────────────────
1. Complex Intersections:
   - Challenge: Multiple converging lanes
   - Observed: Reduced quality in multi-lane scenarios
   - Future work: Multi-line handling

2. Severe Occlusions:
   - Challenge: Vehicles covering road region
   - Observed: Mask generation may include non-road areas
   - Future work: Better mask generation or frame selection

3. Extreme Lighting:
   - Challenge: Strong shadows, nighttime scenes
   - Observed: Reduced inpainting quality under extreme lighting
   - Future work: Lighting-conditional training data

4. Curved Roads:
   - Challenge: Maintaining perspective on curves
   - Observed: Slight quality degradation on curved segments
   - Future work: Curve-specific data augmentation

Note: Systematic failure-rate analysis by scene category is pending.
      Per-video VLM verdicts and numerical metrics can be aggregated
      from the eval_{video_id}.txt and sidecar JSON files.

VLA TRAJECTORY EVALUATION (NEW — ALPAMAYO)
──────────────────────────────────────────
Alpamayo-R1-10B trajectory prediction on edited vs. original videos:

  Metric: minADE (minimum Average Displacement Error, in metres)
  Formula: minADE = min_s (1/T) Σ_t ||pred_xy^(s)_t − gt_xy_t||_2
  Computed over XY only (2D ground plane), T = 64 future waypoints
  
  Protocol:
  1. For each VP-edited video, parse clip_id and camera_name from filename
     Pattern: <clip_id>.<camera_name>_vp_edit_sample0.mp4
  2. Decode last 4 consecutive frames via PyAV; detect FPS → time_step
  3. Stream matching clip from PhysicalAI-AV HuggingFace dataset
     using time_step = 1/fps for temporal alignment
  4. Replace original camera frames with VP-decoded frames
     (resize via bilinear interpolation if dimensions differ)
  5. Build multi-camera input: 7 cameras × 4 frames = 28 images
  6. Run VLA inference (top_p=0.98, temperature=0.6, max_gen_len=256)
  7. Extract S trajectory samples (S × 64 × 3) + chain-of-thought traces
  8. Compute minADE vs ground-truth ego-motion (XY only)
  
  Output artifacts per video (<video_stem>/ subdirectory):
  - <video_stem>_inference.json:
      video_id, clip_id, camera_name, min_ade_meters,
      reasoning_traces (chain-of-thought), temporal_config
      (vp_video_fps, time_step_seconds, num_frames_per_camera),
      metrics (inference_time_seconds, gpu_memory_used/peak_gb,
      ram_used/peak_mb), success flag
  - <video_stem>_vis_data.npz:
      pred_xyz (S,64,3), pred_rot (S,64,3,3),
      gt_future_xyz (64,3), gt_future_rot (64,3,3),
      ego_history_xyz (16,3), ego_history_rot (16,3,3),
      image_frames (N_cam×T,H,W,3), camera_indices (N_cam,)
  - <video_stem>_overlay.mp4 (H.264, CRF=18, yuv420p):
      Trajectory overlay with pinhole projection on camera view
      + BEV inset (250×250 px, bottom-left) + progressive reveal
      + HUD text (minADE, clip_id, camera) + legend box
  
  Aggregate report (<run_id>_report.txt, workflow mode):
  - Total/successful/failed video counts
  - Average inference time per video
  - Average/maximum GPU memory peak (GB)
  - Average/maximum RAM peak (MB)
  - Average/best/worst minADE (metres)
  - Per-video breakdown with status, timings, and clip metadata
  Written to GCS: outputs/alpamayo/<run_id>/<run_id>_report.txt

  CLI mode also produces inference_summary.json (array of all results).

  A higher minADE (compared to unedited baseline) indicates the edited lane
  markings caused the VLA to plan a different trajectory — quantifying the
  perceptual impact of the lane editing on autonomous driving behaviour.

  Visualization Technical Details:
  - Camera projection: ego frame (x=forward, y=left, z=up) →
    camera frame (z=forward, x=right, y=down)
  - Focal length: f_x = f_y = (W/2) / tan(FOV/2)
  - FOV auto-detected: 120° for *120fov*, 30° for *30fov*, 70° otherwise
  - GT drawn in red (thickness=3), predictions in 5 cycling colors
  - BEV display frame: dx=−y, dy=x (forward=up), auto-scaled
  - Ego history drawn in gray, white dot at ego origin
  - Batch mode: render_all_in_directory() scans *_vis_data.npz files

COMMAND-LINE INTERFACES
───────────────────────
0. Master Pipeline (End-to-End):
   bash scripts/build_and_run.sh
   
   # With overrides:
   RUN_ID=003 \
   SAM2_CHUNK_START=0 SAM2_CHUNK_END=9 \
   VP_CAPTION_REFINE_ITERS=5 \
   HF_TOKEN="hf_xxx" \
   bash scripts/build_and_run.sh

1. SAM2 Video Processing:
   python process_videos_sam2.py \
     --input_uris video1.mp4 video2.mp4 \
     --output_dir /path/to/output \
     --max_frames 150

2. Dataset Generation:
   hlx wf run data_generation.fluxfill_data_generation_wf \
     -p source_gcs_prefix="..." \
     -p num_videos=10000 \
     -p chunk_start=0 \
     -p chunk_end=199

3. Dataset Filtering:
   python filter_fluxfill_dataset.py \
     --input_dir gs://[GCS_BUCKET]/dataset \
     --count single \
     --color white \
     --pattern solid

4. LoRA Training:
   python train_fluxfill_inpaint_lora.py \
     --train_data /path/to/train.csv \
     --pretrained_model_path /path/to/flux_inp \
     --output_dir /path/to/output \
     --num_train_epochs 5 \
     --learning_rate 1e-5 \
     --gradient_accumulation_steps 4 \
     --checkpointing_steps 100

   Note: MAX_TRAIN_STEPS can be left empty to train on full dataset for NUM_TRAIN_EPOCHS

5. VideoPainter Editing:
   python infer/edit_bench.py \
     --meta_csv /path/to/meta.csv \
     --video_root /path/to/videos \
     --mask_root /path/to/masks \
     --instruction "lane single white solid" \
     --output_dir /path/to/output \
     --llm_model /path/to/Qwen2.5-VL-7B-Instruct \
     --caption_refine_iters 10 \
     --caption_refine_temperature 0.1 \
     --strength 1.0

   Note: Environment variables can override defaults:
   CAPTION_REFINE_ITERS=5 VP_STRENGTH=0.85 bash scripts/build_and_run.sh

6. Alpamayo VLA Inference (NEW):
   # Standalone:
   python run_inference.py \
     --video_path /path/to/vp_edited_video.mp4 \
     --output_dir /path/to/output \
     --model_id nvidia/Alpamayo-R1-10B \
     --num_traj_samples 1
   
   # Via build script (auto-discovers VP output):
   HF_TOKEN="hf_xxx" bash scripts/build_and_run.sh
   
   # Via HLX workflow:
   hlx wf run workflow_alpamayo.alpamayo_vla_inference_wf \
     --video_data_gcs_path "gs://bucket/outputs/vp/run_001/" \
     --output_run_id "alp_run_001" \
     --model_id "/workspace/alpamayo/checkpoints/alpamayo-r1-10b" \
     --num_traj_samples 1 \
     --video_name "auto"

          dataset

          (bilinear resize to 576×1024)
        - Blacks out non-target cameras per multi-camera strategy
        - Runs Alpamayo inference twice (original vs. generated) under
          identical conditions
        - Computes minADE (minimum Average Displacement Error) against
          ground-truth trajectories for both passes
        - Outputs structured JSON: predicted trajectories, minADE values,
          chain-of-thought reasoning traces, temporal parameters, GPU memory
          (via nvidia-smi), RAM usage (via psutil), inference timing
        - Outputs NPZ: full trajectory tensors (predicted, ground-truth,
          ego-history) for offline visualisation

        - VLAVideoMetrics dataclass for structured per-video results

        - Aggregate report: average, best, worst minADE across all videos,

C.5.3 visualize_video.py (964 lines) — Trajectory visualisation renderer.

      Core functionality:

          x_cam = -y_ego, y_cam = -z_ego, z_cam = x_ego

C.5.5 Dockerfile (105 lines) — Production container.

      Base: nvidia/cuda:12.1.1-cudnn8-devel-ubuntu22.04
      Python: 3.12 (via deadsnakes PPA)
      Key packages: PyTorch 2.8+cu121, flash-attention, transformers,
      accelerate, HLX workflow SDK

      Notable: Applies a sed monkey-patch to the physical_ai_av dataset
      library. The NVIDIA PhysicalAI-AV package's .to_numpy() method returns
      read-only Apache Arrow arrays; scipy.spatial.transform.Rotation
      (called by Alpamayo's geometry.py) attempts in-place modification,
      causing ValueError. The patch wraps .to_numpy() calls with np.array()
      to produce writable copies. This is a runtime bug in the upstream
      dataset library, not in Alpamayo itself.
